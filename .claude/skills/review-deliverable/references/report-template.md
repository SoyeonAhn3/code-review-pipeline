## 5. Methodology

**분석 구조**: Security Agent → Performance Agent → Quality Agent → Summary Agent (교차 검증)

**점수 산출**: 100 - (Critical x 15) - (Warning x 3) - (Info x 0)

**등급 기준**: A(90+) / B(80+) / C(70+) / D(60+) / F(60 미만)

**심각도 기준**:
- **Critical**: 즉시 수정 필요 — 보안 사고, 데이터 손실, 서비스 장애 유발 가능
- **Warning**: 조기 수정 권장 — 운영 안정성, 유지보수성에 영향
- **Info**: 참고사항 — 개선하면 좋지만 당장 문제 없음

## 작성 규칙

### Critical Issues (Section 3)

1. **현재 구조**: 코드 스니펫이 아닌 구조/패턴 수준으로 기술
   - 좋은 예: "API 키를 별도 config.py에 하드코딩하여 import하는 구조"
   - 나쁜 예: "`from config import API_KEY`"

2. **개선 방향**: 어떻게 바꾸어야 하는지 구조 수준으로 기술
   - 좋은 예: ".env 파일에서 os.environ.get()으로 런타임 주입, .gitignore 적용"
   - 나쁜 예: "`os.environ.get('API_KEY')`"

3. **난이도 기준**:
   - **하**: 단순 수정, 30분 이내 (변수 변경, 가드 조건 추가 등)
   - **중**: 구조 변경 필요, 1~3시간 (함수 분리, 패턴 변경 등)
   - **상**: 아키텍처 영향, 4시간+ (async 전환, 클래스 재설계 등)

### Warning Issues (Section 4)

1. 리뷰 원본의 배치별 구분을 **유형별**(보안/성능/코드 품질)로 재분류
2. Before/After 없이 이슈 요약 + 개선 방향 1줄
3. 복합 이슈(보안+성능 등)는 주된 카테고리에 배치
