1. 프로젝트 개요
================================================================================

[프로젝트명]
AI Code Review Pipeline — Multi-Agent 코드 리뷰 자동화 시스템

[한 줄 설명]
코드를 넣으면 AI Agent 4명이 보안 → 성능 → 코드 품질 → 종합 순서로
리뷰하고, 점수와 수정 제안이 포함된 리포트를 생성하는 시스템.

[개발 배경]
AI를 활용해 다양한 업무 자동화 프로그램을 개발하는 과정에서,
생성된 코드가 작동은 하지만 아래와 같은 문제가 반복적으로 발생함을 확인함:
  - 비효율적 코드: 불필요한 반복, 최적화되지 않은 로직
  - 중복 코드: 같은 기능이 여러 곳에 복사-붙여넣기 형태로 존재
  - 위험한 코드: 보안 취약점, 에러 핸들링 누락, 하드코딩된 민감 정보
코드가 "돌아간다"는 것만으로는 충분하지 않으며,
품질·보안·성능까지 챙기려면 체계적인 리뷰가 필요함.
그러나 매번 수동으로 검토하기엔 시간과 관점의 한계가 있어,
이를 AI Agent가 자동으로 수행하는 파이프라인을 만들게 됨.

[왜 만드는가]
코드 리뷰는 모든 개발팀에서 하는 작업이지만, 사람이 하면 시간이 걸리고
관점이 편향될 수 있음. 보안 전문가는 성능을 놓치고, 시니어 개발자는
코드 스타일을 덜 볼 수 있음. 여러 전문가 관점을 AI Agent로 분리하면
빠짐없이, 빠르게, 일관된 기준으로 리뷰할 수 있음.

[포트폴리오에서 증명하는 것]
- Sequential Pipeline 오케스트레이션 (Agent 간 컨텍스트 전달)
- Claude API Tool Use (GitHub API 연동)
- 프롬프트 엔지니어링 (역할별 전문가 페르소나)
- 구조화된 출력 (JSON 형식 리뷰 결과)

[대상 사용자]
- 개발자: PR 올리기 전 셀프 리뷰
- 팀 리드: 리뷰 기준 일관성 확보
- 주니어 개발자: 코드 품질 학습 도구


================================================================================
2. 시스템 아키텍처
================================================================================

[전체 흐름]
  사용자 입력 (코드 텍스트 또는 GitHub PR URL)
    → Orchestrator (순서 제어 + 컨텍스트 관리)
      → Agent 1: Security (보안 취약점 탐지)
      → Agent 2: Performance (성능 병목 탐지)
      → Agent 3: Code Quality (코드 품질 + 가독성)
      → Agent 4: Summary (종합 + 점수 + 수정 제안)
    → 결과 출력 (Markdown 리포트 / Streamlit UI)

[설계 원칙]

1. 순차 실행 + 컨텍스트 누적
   각 Agent가 이전 Agent의 발견 사항을 함께 받음.
   이유: Performance Agent가 Security 결과를 보면
   "보안 수정이 성능에 미치는 영향"을 분석할 수 있음.
   Quality Agent가 이전 두 결과를 보면 이미 지적된 건 스킵하고
   새로운 관점에만 집중할 수 있음.

2. 같은 LLM, 다른 페르소나
   4개 Agent 전부 Claude API를 호출하지만 시스템 프롬프트가 다름.
   이유: 별도 모델이나 서비스를 쓸 필요 없이,
   프롬프트만으로 전문가 역할을 분리할 수 있음.

3. Tool Use 기반 구조화 출력
   각 Agent의 출력을 Claude API의 tool_use 기능으로 강제.
   이유: 프롬프트로 "JSON으로 답해줘"라고 하면 가끔 형식이 깨지지만,
   tool_use + tool_choice를 쓰면 정해진 스키마에 맞춰서만 응답함.
   → JSON 파싱 실패 확률이 거의 0.
   방법: 각 Agent마다 전용 tool 스키마를 정의하고,
   tool_choice={"type": "tool", "name": "report_xxx_issues"}로 강제 호출.
   response.content[0].input에서 바로 dict를 꺼내 쓸 수 있음.

4. Agent 간 교차 반론 (Cross-Review)
   후행 Agent가 선행 Agent의 제안에 동의/반대 의견을 제시.
   이유: 보안 수정이 성능을 해치거나, 성능 최적화가 가독성을 해치는 경우를
   자동으로 잡아낼 수 있음. 단순 전달이 아닌 "검증"이 추가되어 리뷰 품질 향상.
   방법: 각 Agent 출력 스키마에 cross_review 필드를 추가.
   프롬프트에 "이전 Agent의 수정 제안이 당신의 관점에서 문제가 없는지
   검토하고, 동의/주의/반대 의견을 제시하세요" 지시.


================================================================================
3. Agent 상세 설계
================================================================================

[Agent 1: Security Agent]

  역할: 보안 취약점 탐지
  확인 항목:
    - SQL Injection 가능성 (문자열 결합으로 쿼리 생성)
    - XSS 취약점 (사용자 입력을 필터 없이 렌더링)
    - 하드코딩된 비밀번호/API 키
    - 안전하지 않은 API 사용 (eval, exec, pickle.loads 등)
    - 인증/권한 체크 누락
    - 의존성 보안 (알려진 취약 버전)

  출력 형식:
  {
    "agent": "security",
    "issues": [
      {
        "severity": "critical" | "warning" | "info",
        "confidence": "high" | "medium" | "low",
        "line": 42,
        "code_snippet": "query = f\"SELECT * FROM users WHERE id = {user_id}\"",
        "issue": "SQL Injection: 사용자 입력을 직접 쿼리에 삽입",
        "suggestion": "파라미터 바인딩 사용: cursor.execute('SELECT ... WHERE id = ?', (user_id,))"
      }
    ],
    "summary": "보안 이슈 2건 (critical 1, warning 1)"
  }


[Agent 2: Performance Agent]

  역할: 성능 병목 탐지
  확인 항목:
    - N+1 쿼리 패턴 (루프 안에서 DB 호출)
    - 불필요한 중첩 루프 (O(n²) 이상)
    - 메모리 누수 가능성 (큰 리스트를 계속 append)
    - 블로킹 호출 (동기 I/O in async 코드)
    - 캐시 미활용
    - 불필요한 데이터 로딩 (전체 조회 후 필터)

  입력: 코드 + Security Agent 결과
  이유: "보안 수정으로 추가될 검증 로직이 루프 안에 들어가면
    성능 영향이 있을 수 있다"는 교차 분석 가능.

  출력 형식: Security와 동일 구조 (agent: "performance")
  + cross_review 필드 추가:
  "cross_review": [
    {
      "target_agent": "security",
      "target_issue": "해당 이슈 요약",
      "opinion": "agree" | "caution" | "disagree",
      "comment": "성능 관점에서의 의견"
    }
  ]


[Agent 3: Code Quality Agent]

  역할: 코드 품질 및 가독성
  확인 항목:
    - 네이밍 컨벤션 (변수명, 함수명)
    - 함수 길이 / 복잡도 (Cyclomatic Complexity 추정)
    - DRY 위반 (중복 코드)
    - 타입 힌트 누락 (Python의 경우)
    - 에러 핸들링 누락 (bare except, 빈 catch)
    - 주석/문서화 부족
    - 매직 넘버 사용

  입력: 코드 + Security 결과 + Performance 결과
  이유: 이전 Agent가 이미 지적한 라인은 중복 지적하지 않음.

  출력 형식: 동일 구조 (agent: "quality")
  + cross_review 필드 추가 (Security, Performance 제안에 대한 품질 관점 의견)


[Agent 4: Summary Agent]

  역할: 전체 종합, 중복 제거, 점수 산정, 수정 우선순위 제안
  입력: 코드 + 3개 Agent 결과 전부

  처리:
    1. 중복 이슈 제거 (같은 라인, 비슷한 지적)
    2. 교차 반론 종합 (Agent 간 의견 충돌 시 최종 판단)
       - caution/disagree 의견이 있으면 양쪽 근거를 비교하여 결론 도출
       - "보안+성능 트레이드오프" 등 복합 이슈로 태깅
    3. 전체 점수 산정 (100점 만점)
       - critical 이슈 1건당 -15점
       - warning 1건당 -5점
       - info 1건당 -1점
    4. 수정 우선순위 Top 3 선정 (impact x effort 기준)
    5. 종합 코멘트 생성

  출력 형식:
  {
    "overall_score": 72,
    "grade": "C+",
    "total_issues": {"critical": 1, "warning": 3, "info": 5},
    "top_3_actions": [
      {"priority": 1, "issue": "SQL Injection on line 42", "effort": "5분"},
      {"priority": 2, "issue": "N+1 query in user_list()", "effort": "30분"},
      {"priority": 3, "issue": "Missing error handling in api_call()", "effort": "15분"}
    ],
    "cross_review_conflicts": [
      {
        "issue": "line 80: 입력값 검증 추가 (Security 제안)",
        "conflict": "Performance Agent가 루프 내 반복 실행 시 성능 저하 경고",
        "resolution": "루프 밖에서 1회만 검증하도록 수정 권장"
      }
    ],
    "comment": "보안 이슈 1건이 가장 시급합니다. ...",
    "all_issues": [...]
  }


================================================================================
4. 오케스트레이터 설계
================================================================================

[핵심 역할]
  1. Agent 호출 순서 관리 (Security → Performance → Quality → Summary)
  2. 이전 Agent 결과를 다음 Agent 프롬프트에 포함
  3. 각 Agent의 Tool Use 출력 수신 + 검증
  4. 교차 반론 충돌 감지 및 Summary Agent에 전달
  5. 에러 발생 시 해당 Agent 스킵 + 나머지 계속 진행

[상태 관리]
  review_state = {
    "code": "...(원본 코드)...",
    "language": "python",
    "findings": {
      "security": {...},
      "performance": {...},
      "quality": {...},
      "summary": {...}
    },
    "errors": []
  }

[언어 자동 감지]
  코드 첫 줄에서 파일 확장자 또는 문법 패턴으로 감지.
  이유: Agent 프롬프트에 "이 코드는 Python입니다"를 포함해야
  언어별 보안 취약점을 정확히 찾음.
  감지 실패 시: 사용자에게 언어 선택 요청.


================================================================================
5. 입력 방식
================================================================================

[방식 1: 코드 직접 입력] ★ MVP
  Streamlit 텍스트 영역에 코드를 붙여넣기.
  가장 간단하고 외부 API 의존성 없음.

[방식 2: GitHub PR URL] (확장)
  GitHub API로 PR의 diff를 가져옴.
  변경된 파일만 리뷰 (전체 파일이 아닌 diff 기준).

  GitHub API 호출:
    GET /repos/{owner}/{repo}/pulls/{pr_number}/files
    → 변경된 파일 목록 + 각 파일의 patch(diff) 반환

  인증: GitHub Personal Access Token (공개 repo는 불필요)

[방식 3: 파일 업로드] (확장)
  .py, .js, .ts 파일을 직접 업로드.
  여러 파일이면 파일별로 리뷰 후 종합.

6. 출력 방식
================================================================================

[Streamlit 웹 UI] ★ MVP
  왼쪽: 코드 입력 영역
  오른쪽: 리뷰 결과 (탭으로 Agent별 분리)
    - Overview 탭: 전체 점수, Top 3 수정 사항

  다운로드 가능한 .md 파일.
  GitHub에 직접 올리거나 Notion에 붙여넣기 가능.

8. 리스크 및 대처
================================================================================

[리스크 1: 큰 코드의 토큰 제한]

  상황: PR이 2,000줄 이상이면 한 번에 보내기 어려움.
    코드 + 이전 Agent 결과 + 프롬프트를 합치면 토큰 부족 가능.
  영향: Agent가 코드 전체를 보지 못하면 교차 참조 분석이 약해짐.
  대처:
    1차: 파일별로 분할하여 독립 리뷰.
    2차: 변경된 줄(diff)만 추출하여 변경분만 리뷰.
    3차: 코드 요약 Agent를 앞에 추가하여 전체 구조 요약 후
         함수/클래스 단위로 분할 리뷰.
  MVP 범위: 500줄 이하 코드만 지원. 초과 시 사용자에게 안내.


[리스크 2: Agent 간 중복 지적]

  상황: Security가 "입력 검증 없음", Quality도 "에러 핸들링 없음".
    같은 줄, 비슷한 문제를 두 번 지적.
  영향: 리뷰 결과가 장황하고 반복적으로 느껴짐.
  대처:
    1. 각 Agent 프롬프트에 "이전 Agent가 지적한 이슈는 건너뛰세요" 명시.
    2. Summary Agent가 중복 제거.
       같은 라인 + 유사 키워드 → 하나로 병합.
    3. 병합 시 "보안+품질 복합 이슈"로 태깅하여 오히려 가치를 높임.


[리스크 3: AI 오탐 (false positive)]

  상황: AI가 "보안 취약점"이라고 했는데 실제로는 아닌 경우.
    예: test 코드의 하드코딩 값을 "비밀번호 노출"로 오탐.
  영향: 오탐이 많으면 사용자가 리뷰 결과를 신뢰하지 않음.
  대처:
    1. 각 이슈에 confidence 점수 추가 (high/medium/low).
       "확실한 이슈"와 "의심 수준"을 구분.
    2. 프롬프트에 "확실하지 않은 건 confidence: low로 표시하고,
       왜 의심하는지 이유를 설명하세요" 지시.
    3. Summary에서 low confidence 건은 "참고 사항"으로 분류.


[리스크 4: 언어/프레임워크별 전문성 차이]

  상황: Python 리뷰는 잘하는데 Rust나 Go는 약할 수 있음.
  영향: 특정 언어에서 뻔한 취약점을 놓침.
  대처:
    1. MVP: Python, JavaScript/TypeScript만 공식 지원.
       다른 언어는 "정확도가 낮을 수 있음" 안내.
    2. Agent 프롬프트에 언어별 체크리스트 포함.
       Python: pickle, eval, __import__
       JS: innerHTML, document.write, dangerouslySetInnerHTML
    3. 감지된 언어를 프롬프트에 명시.


[리스크 5: Claude API 비용]

  상황: 1회 리뷰에 API 4회 호출. 코드가 길면 토큰 비용 증가.
  대처:
    1. Sonnet 모델 사용 (Opus 대비 비용 약 1/5).
    2. 캐싱: 같은 코드를 다시 리뷰하면 이전 결과 반환 (해시 기반).
    3. 예상 비용: 500줄 코드 1회 리뷰 ≈ $0.02~0.04.
       월 100회 리뷰 ≈ $2~4.


[리스크 6: Agent 출력 형식 불일치] → Tool Use로 근본 해결

  상황: Agent가 요청한 JSON 형식을 정확히 안 따름.
  대처 (변경):
    1차 (근본 해결): Claude API tool_use + tool_choice로 출력 스키마 강제.
       → Agent가 정해진 tool 스키마 외의 형식으로 응답할 수 없음.
       → response.content[0].input에서 바로 dict 추출, 파싱 불필요.
    2차 (폴백): 드물게 tool_use가 실패하는 경우
       텍스트 전체를 "raw_output"으로 저장.
       Summary Agent에게 "비구조화 출력에서 이슈를 추출해주세요" 지시.
    3차: 에러 횟수를 기록하여 스키마/프롬프트 개선에 활용.


================================================================================
9. 평가/검증 전략 (Eval Set)
================================================================================

[목적]
  시스템이 실제로 문제를 잘 잡는지 숫자로 증명하는 체계.
  "만들고 끝"이 아니라 "검증까지 완료"를 포트폴리오에서 보여줌.

[Eval Set 구성]
  의도적으로 문제를 심어둔 샘플 코드 8~10개 + 정답지(expected.json).

  eval_samples/
  ├── sample_01_sql_injection.py        # SQL Injection + 하드코딩 비번
  ├── sample_01_expected.json           # 정답지: 찾아야 할 이슈 목록
  ├── sample_02_n_plus_one.py           # N+1 쿼리 + 불필요한 전체 로딩
  ├── sample_02_expected.json
  ├── sample_03_xss_react.jsx           # dangerouslySetInnerHTML + innerHTML
  ├── sample_03_expected.json
  ├── sample_04_bad_naming.py           # 변수명 a,b,c + 매직 넘버 + 긴 함수
  ├── sample_04_expected.json
  ├── sample_05_mixed_issues.py         # 보안+성능+품질 복합
  ├── sample_05_expected.json
  ├── sample_06_clean_code.py           # 문제 없는 코드 (오탐 테스트용)
  ├── sample_06_expected.json
  ├── sample_07_tricky_false_positive.py # 테스트 코드의 하드코딩 (오탐하면 안 됨)
  ├── sample_07_expected.json
  ├── sample_08_async_blocking.py       # async 안에서 동기 I/O
  ├── sample_08_expected.json
  └── run_eval.py                       # 자동 채점 스크립트

[정답지 형식 (expected.json)]
  {
    "must_find": [
      {"type": "security", "keyword": "sql injection", "line": 2},
      {"type": "security", "keyword": "hardcoded password", "line": 9},
      {"type": "performance", "keyword": "n+1", "line": 5}
    ],
    "should_not_find": [
      {"type": "security", "line": 7, "reason": "정상 코드인데 오탐하면 안 됨"}
    ]
  }

[측정 지표 — 3가지]
  1. 정탐률 (Recall): 심어둔 이슈 중 실제로 찾은 비율 (목표: 80% 이상)
  2. 오탐률 (False Positive Rate): 없는 문제를 있다고 한 건수 (목표: 샘플당 평균 1건 이하)
  3. 교차 반론 유효율: cross_review 의견이 실제로 타당한 비율 (수동 확인)

[자동 채점 로직 (run_eval.py)]
  1. eval 샘플 전체를 파이프라인에 순차 실행
  2. 실제 결과 vs expected.json 비교
     - must_find 항목: keyword가 실제 이슈에 포함되어 있으면 정탐
     - should_not_find 항목: 해당 라인에 이슈가 없으면 정상
  3. 결과를 테이블로 출력

  결과 출력 예시:
  ┌──────────────────────┬────────┬────────┬──────────┐
  │ 샘플                 │ 정탐률 │ 오탐수 │ 교차반론 │
  ├──────────────────────┼────────┼────────┼──────────┤
  │ sql_injection        │ 100%   │ 0      │ -        │
  │ n_plus_one           │ 100%   │ 1      │ 유효     │
  │ xss_react            │ 100%   │ 0      │ -        │
  │ bad_naming           │ 75%    │ 0      │ -        │
  │ mixed_issues         │ 80%    │ 1      │ 유효     │
  │ clean_code           │ -      │ 0      │ -        │
  │ tricky_false_pos     │ -      │ 0      │ -        │
  │ async_blocking       │ 100%   │ 0      │ 유효     │
  ├──────────────────────┼────────┼────────┼──────────┤
  │ 전체 평균            │ 87.5%  │ 0.5건  │ 80%      │
  └──────────────────────┴────────┴────────┴──────────┘

[프롬프트 튜닝 사이클]
  eval 결과를 기반으로 프롬프트를 반복 개선.
  "프롬프트 v1: 정탐 75% → v2: 정탐 87% → v3: 정탐 92%"
  이 개선 과정 자체가 포트폴리오에서 어필할 수 있는 포인트.

  스킬                        적용 위치

  Structured Output           Tool Use 기반 스키마 강제 + 폴백

  Error Handling              Agent 스킵 + 폴백 + 로깅
  Streamlit                   인터랙티브 웹 UI
  코드 분석 도메인              보안/성능/품질 체크리스트 설계

    "같은 LLM을 프롬프트만으로 4명의 전문가로 분리하고,
     이전 Agent의 결과를 다음 Agent가 참고하면서
     교차 분석이 가능한 구조를 설계했습니다.
     단순 전달이 아니라 Agent가 서로의 제안을 검증(교차 반론)하여
     보안 수정이 성능을 해치는 경우까지 자동으로 잡아냅니다.
     Tool Use로 출력 형식을 강제해 파싱 실패율을 0에 가깝게 낮추었고,
     8개 Eval 샘플로 정탐률/오탐률을 정량 측정하여
     프롬프트를 반복 개선한 과정까지 포함됩니다."
